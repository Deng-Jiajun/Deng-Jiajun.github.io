1. 简介

1.1 Markdown是什么？

Markdown是一种可以使用普通文本编辑器编写的标记语言，通过简单的标记语法，它可以使普通文本内容具有一定的格式。 Markdown具有一系列衍生版本，用于扩展Markdown的功能（如表格、脚注、内嵌HTML等等），这些功能原初的Markdown尚不具备，它们能让Markdown转换成更多的格式，例如LaTeX，Docbook。Markdown增强版中比较有名的有Markdown Extra、MultiMarkdown、 Maruku等。这些衍生版本要么基于工具，如Pandoc；要么基于网站，如GitHub和Wikipedia，在语法上基本兼容，但在一些语法和渲染效果上有改动。

——百度百科

3.3 文档图片集成

Markdown 文档本身只是一个文本文件，对于图片的加载，依靠图片路径载入，而无法与文档集成。 这里提供几种对于文档图片集成问题的解决思路及优缺点分析：

3.3.1 在线图床

Markdown 文档中使用在线图床是网络上比较通用的一种解决方案，在线图床也有各种现成可用的解决方案，保证了文章图片的访问。但是在线图床不论是免费或是收费，服务总是存在不确定性，且受到网络质量的影响，整体配置过程也较繁琐，一旦要迁移，体验可能不太愉快。

提供一种不使用平台图床的解决方案（需要有云服务器）： 本地所有的图片保存到 img 统一路径下，markdown 文档中需要提供云端访问时，图片拖到云主机可访问的img路径下，将本地 markdown 中的路径全局替换为云主机的img访问地址。如果要换主机 ip 或者域名，统一全局替换就行。

3.3.2 图片转 base64

示例图片（无特殊含义）：

一种可以称得上「奇技淫巧」的方案，将图片转为 base64 编码，通过markdown 或 html <img>标签直接嵌入文档中，导出为 pdf 和 docx 文件时也能正常显示，但缺点也很明显：

  • 首先，文档体积急剧膨胀，图片转 base64 编码会在原文件的基础上膨胀约 30%，当然，一般来说，文档图片膨胀 30% 并不是无法接受的。
  • 其次，base64 编码在 Markdown 文档中显示效果及其不友好，一张 12k 的图片，在文档中占到 8 屏。虽然可以通过在文档末尾添加图片列表，但是一旦图片数量较多，文档维护难度大。

相对来说，图片转 base64 算得上是一种折衷的好方法，对于只能通过 .md 文本传输的场景，是唯一解，相较图床而言不依赖网络环境，并且图片集成在文档中，永久保存，只要阅读器正常渲染，文本与图片就是一体的，建议在图片量不大且要求明确的文档中使用。

3.3.3 随附图片文件夹

既然本身就是文档和图片分离的，最简单的方式自然是直接把图片文件夹也交付给对方，只要在编写文档时使用相对路径，那么将 .md 文档和对应的图片文件夹一共交付即可。缺点很明确，不是单文件，需要打包提供。

3.3.4 转换通用格式文档

以上3个方案的前提都是需要交付 .md 格式文档，且对方有markdown文档阅读器（最好还是同一个软件）。那么最直接避免格式不同造成不必要麻烦的方式，就是直接转换为通用格式文档，.docx 或 .pdf。

除非只能提供.docx，否则 .pdf 是更好的选择，一是 .pdf 可以保持显示效果的一致性，二是 .docx 对一些样式不支持或不一致，无法显示效果，如引用和表格。

这两种文档保证了文档和图片的集成，是通用的文档格式，但在转换为这两种格式保证可阅读性的同时，也关闭了对其进行修改的可能性。（.docx 转换为 .md 效果不理想）

实际上 markdown 本身作为一种简单语法的标记语言，图片只是附加的内容，markdown本身要做的就是内容与样式的分离，做出修改本质上也是对文本的修改。实际上大多数情况下要修改的，多半也是文字内容而不是修改图片，那么只要做到不必要的时候不引入、少引入图片，交付提供 pdf，该问题就自然被绕过了。

4. 总结

Markdown 的设计逻辑是内容与样式分离，只要形成自己的书写习惯与规范，就不需要关注显示效果上的问题，减轻了文档编写的负担，使编写者进入所谓「心流」的状态，专注于内容本身，而不是显示效果、样式等内容。Word 这类文字处理软件，功能复杂强大，但同时也很容易将使用者的注意力分散到各个细节的处理上，影响了文档编写的效率。使用简单，还算美观的显示效果，这是个人认为使用 Markdown + Typora 这一组合编写文档的优势所在，习惯之后能够轻松构建一份合格的文档。
